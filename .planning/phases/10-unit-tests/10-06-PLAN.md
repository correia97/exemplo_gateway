Repository base class (src/OpenCode.Domain/Implementations/Repository.cs):
```csharp
public class Repository<T> : IRepository<T> where T : class
{
    protected readonly DbContext Context;
    protected readonly DbSet<T> DbSet;
    public Repository(DbContext context) { ... }
    Task<T?> GetByIdAsync(int id, ...) => DbSet.FindAsync([id], ...);
    Task<PagedResult<T>> GetAllAsync(int page, int pageSize, ...) { ... }  // Base impl
    Task<T> AddAsync(T entity, ...) { DbSet.AddAsync; SaveChangesAsync; return entry.Entity; }
    Task UpdateAsync(T entity, ...) { Entry(entity).State = Modified; SaveChangesAsync; }
    Task DeleteAsync(T entity, ...) { DbSet.Remove; SaveChangesAsync; }
}
```

  IMPORTANT: `HasDefaultSchema("dragonball")` in OnModelCreating will cause InMemory to fail (InMemory doesn't support schemas). The test must create DragonBallContext with an action that CLEARS the default schema configuration. Use this approach:

  ```csharp
  private static DragonBallContext CreateContext()
  {
      var options = new DbContextOptionsBuilder<DragonBallContext>()
          .UseInMemoryDatabase(Guid.NewGuid().ToString())
          .Options;
      var context = new DragonBallContext(options);
      // InMemory provider doesn't support schemas - EF Core migrations/SQL features
      // that use HasDefaultSchema will throw in InMemory. We must ensure the DbContext
      // is configured without schema requirements for test purposes.
      // DragonBallContext.OnModelCreating calls HasDefaultSchema("dragonball") which
      // causes issues with InMemory. The context constructor must handle this.
      // As a workaround, ensure the DB is created:
      context.Database.EnsureCreated();
      return context;
  }
  ```

  Actually, `HasDefaultSchema` sets the schema metadata — it should NOT cause InMemory to fail. `HasDefaultSchema` is metadata-only. EF Core InMemory ignores schema settings. So this should work fine with the standard DragonBallContext constructor. Only `.ToTable("TableName", "schema")` with explicit schema causes InMemory issues. Since the context uses `HasDefaultSchema("dragonball")` (not explicit schema on ToTable), InMemory should handle it. Use the standard constructor directly.

  But `HasDefaultValueSql("NOW()")` WILL cause InMemory to fail because InMemory does not support SQL expressions. For the CreatedAt and UpdatedAt properties, InMemory will use the default value for DateTime (DateTime.MinValue) instead of NOW(). This is acceptable for test purposes — the tests should NOT assert on these values unless they're explicitly set.

  So create context directly:
  ```csharp
  var options = new DbContextOptionsBuilder<DragonBallContext>()
      .UseInMemoryDatabase(Guid.NewGuid().ToString())
      .Options;
  var context = new DragonBallContext(options);
  context.Database.EnsureCreated();
  ```
