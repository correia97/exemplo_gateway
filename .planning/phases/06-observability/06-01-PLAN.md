<action>
Add `opentelemetry` to the plugin list in config.yaml. The plugin is already whitelisted for `prometheus` — append `opentelemetry` after `prometheus`. The `opentelemetry` plugin sends traces to the Jaeger collector via OTLP HTTP.

Create `init-routes-otel.sh` — a new Admin API script that creates a **global rule** for the `opentelemetry` plugin. This script is separate from Phase 5's `init-routes.sh` to maintain clear phase ownership.

The script should:
1. Wait for Kong Admin API to be ready (port 9180 responds with 200)
2. Create a global rule named `opentelemetry-tracing` that configures:
   - `collector.address`: `http://jaeger:4318/v1/traces` (Kong runs in container, resolves Jaeger via container name)
   - `collector.request_timeout`: 1 (second — avoid startup delays per Pitfall 10)
   - `batch.batch_timeout`: 1 (second)
   - `batch.max_queue_size`: 1024
   - `additional_attributes`: `service.name` = `apisix-gateway`
3. The script should share the same `ADMIN_API` and `ADMIN_KEY` variables as init-routes.sh

Use the same script patterns as init-routes.sh (bash with curl, error handling). The script can be standalone — the AppHost will call both init scripts sequentially.

**Key technical details:**
- Kong 3.9.x OTel plugin supports OTLP HTTP on port 4318
- The plugin must be whitelisted in config.yaml first or Admin API will reject it with 400
- The global rule applies to ALL routes automatically (both GET and write routes)
- traceparent propagation is automatic when OTel plugin is enabled — Kong reads incoming traceparent and creates child spans

Modify `src/OpenCode.AppHost/Program.cs` to:

1. **Add Jaeger container** after the Kong section (before `builder.Build().Run()`):
   ```csharp
   var jaeger = builder.AddContainer("jaeger", "jaegertracing/all-in-one:latest")
       .WithEnvironment("COLLECTOR_OTLP_ENABLED", "true")
       .WithEndpoint(port: 4318, targetPort: 4318, scheme: "http", name: "otlp-http")
       .WithEndpoint(port: 16686, targetPort: 16686, scheme: "http", name: "ui");
   ```
   - Port 4318: OTLP HTTP receiver (Kong and .NET APIs send traces here)
   - Port 16686: Jaeger UI for viewing traces
   - `COLLECTOR_OTLP_ENABLED=true`: Enables native OTLP receiver in all-in-one image
   - Jaeger does NOT need to wait for any service — it can start independently

2. **Set OTLP endpoint env vars on both API projects** (add after existing `.WithReference(postgres)` call chains):
   ```csharp
   // On dragonballApi:
   .WithEnvironment("OTEL_EXPORTER_OTLP_ENDPOINT", "http://host.docker.internal:4318/")
   
   // On musicApi:
   .WithEnvironment("OTEL_EXPORTER_OTLP_ENDPOINT", "http://host.docker.internal:4318/")
   ```
   - `host.docker.internal` because API projects run as host processes (not containers) in Aspire inner-loop
   - The trailing `/` is required by OTLP HTTP protocol
   - Per D-05: Explicit env vars override the auto-configured Aspire Dashboard endpoint (which doesn't exist since DCP isn't installed)

3. **Update the Kong `WithArgs`** to chain both init scripts sequentially. The AppHost already mounts `.planning/phases/05-apisix-gateway` as `/scripts` in the Kong container. Since `init-routes-otel.sh` will be placed alongside `init-routes.sh` in that directory, both scripts are available.

   Change the `WithArgs` to:
   ```
   chmod +x /scripts/init-routes.sh && /scripts/init-routes.sh && chmod +x /scripts/init-routes-otel.sh && /scripts/init-routes-otel.sh && exec /usr/local/openresty/bin/openresty -p /usr/local/Kong -g 'daemon off;'
   ```

   This ensures:
   - `init-routes.sh` (Phase 5 — routes, request-id, Keycloak validation) runs first
   - `init-routes-otel.sh` (Phase 6 — OTel global rule) runs second
   - OpenResty starts last, after both configs are applied
   - No additional bind mounts needed
