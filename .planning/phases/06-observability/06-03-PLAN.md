**Scenario 1: Jaeger is running and accepting traces**
```bash
# Check Jaeger health
curl -s http://localhost:16686/ | Select-String -Pattern "Jaeger UI"

# Check Jaeger API services endpoint (lists all services sending traces)
curl -s http://localhost:16686/api/services | ConvertFrom-Json | ConvertTo-Json
```
Expected: Jaeger UI page returns HTML. API returns `{"data":["apisix-gateway","dragonball-api","music-api"]}` (or similar).

**Scenario 2: Kong generates OTel spans**
```bash
# Send a request through Kong to DragonBall API
curl -s http://localhost:9080/dragonball/api/characters -H "Authorization: Bearer $(TOKEN)"

# Query Jaeger for traces from apisix-gateway service
curl -s "http://localhost:16686/api/traces?service=apisix-gateway&limit=1" | ConvertFrom-Json | ConvertTo-Json -Depth 10
```
Expected: At least one trace returned. The span shows `http.url`, `http.method`, service name `apisix-gateway`.

**Scenario 3: .NET API traces appear with Correlation ID**
```bash
# Send a request with a specific correlation ID
curl -s http://localhost:9080/dragonball/api/characters -H "X-Correlation-Id: test-correlation-otel-verification" -H "Authorization: Bearer $(TOKEN)"

# Query Jaeger for traces from dragonball-api
curl -s "http://localhost:16686/api/traces?service=dragonball-api&limit=1" | ConvertFrom-Json | ConvertTo-Json -Depth 10
```
Expected: Trace returned. Span tags include `http.request_id` or show the correlation ID in span attributes.

**Scenario 4: Database query spans are visible**
```bash
# Send a request that triggers a database query (get a specific character)
curl -s http://localhost:9080/dragonball/api/characters/1 -H "Authorization: Bearer $(TOKEN)"

# Query Jaeger for traces from dragonball-api, filter to most recent
curl -s "http://localhost:16686/api/traces?service=dragonball-api&limit=5" | ConvertFrom-Json | ConvertTo-Json -Depth 10
```
Expected: Within the trace, there is a child span with `db.system=postgresql` and `db.statement` containing the SQL query text (sanitized, no parameter values).

**Scenario 5: Complete trace chain (Kong → .NET API → PostgreSQL)**
Wait for a trace that has spans from ALL three services. Use the Jaeger API to find a trace with spans from multiple services:

```bash
# First get list of recent traces from apisix-gateway
$traces = curl -s "http://localhost:16686/api/traces?service=apisix-gateway&limit=1" | ConvertFrom-Json
$traceId = $traces.data[0].traceID

# Then fetch the full trace
curl -s "http://localhost:16686/api/traces/$traceId" | ConvertFrom-Json | ConvertTo-Json -Depth 10

# Or open in browser: http://localhost:16686/trace/$traceId
```
Expected: A single trace contains:
- Root span: `apisix-gateway` (HTTP reverse proxy)
- Child span: `dragonball-api` (ASP.NET Core)
- Child span: `postgresql` (Npgsql database query)
- The spans are linked via `traceparent` / W3C trace context

**Scenario 6: Correlation ID appears in structured logs**
This requires checking the API's console output when running in Aspire. Since the AppHost is compile-only, the API processes run on the host. Check the process console output:

```bash
# Alternative: Send a request and check application logs via OTel
# The Correlation ID is exported as a log attribute to Jaeger
# In Jaeger UI: Search for a trace from dragonball-api
# Expand the span -> look for "Logs" tab or span events
# The Correlation ID should appear in the log scope

# Programmatic check: query trace details and look for correlationId tag
curl -s "http://localhost:16686/api/traces?service=dragonball-api&limit=5" | 
    Select-String -Pattern "correlationId|correlation_id|CorrelationId"
```
Also provide manual Jaeger UI steps:
1. Open http://localhost:16686
2. Select service: `dragonball-api`
3. Click "Find Traces"
4. Click into a trace
5. Look for span attributes containing `correlationId`
