Admin API global rule format:
  PUT /global_rules/correlation-id
  Body: {
    "plugins": {
      "request-id": {
        "header_name": "X-Correlation-Id",
        "include_in_response": true,
        "algorithm": "uuid"
      }
    }
  }

Global rules have higher priority than per-route plugins and apply to ALL routes.
The header is set in the request to upstream AND in the response to client.

Per CONTEXT.md D-03: "request-id global rule" — must be global, not per-route.
Per RESEARCH.md: Global rule must be created BEFORE routes to apply to all requests.
In practice: APISIX evaluates global rules at request time, so creation order during
startup doesn't matter as long as the global rule exists before traffic arrives.
But for safety, create it first in the init script before upstreams and routes.

Per CONTEXT.md D-02: "Header name: X-Correlation-Id (consistent with .NET
UseCorrelationId() middleware)"

## Tasks

### Task 1: Add request-id global rule to init-routes.sh
Add the global rule creation as the FIRST Admin API call in `init-routes.sh`, right after the Wait loop and before upstream/route creation. This ensures the global rule is registered before any route configuration.

Insert after the `echo "Admin API ready."` line:

```bash
# === Create Global Rules (applied to ALL routes) ===

echo "Creating global rule: request-id (X-Correlation-Id)"
curl -s -X PUT "$ADMIN_API/global_rules/correlation-id" \
    -H "X-API-KEY: $ADMIN_KEY" \
    -H "Content-Type: application/json" \
    -d "$(cat <<EOF
{
    "plugins": {
        "request-id": {
            "header_name": "X-Correlation-Id",
            "include_in_response": true,
            "algorithm": "uuid"
        }
    }
}
EOF
)"
```

**Why first:** While APISIX evaluates all rules at request time (not creation time), placing the global rule first in the script makes the dependency explicit — the correlation ID is foundational infrastructure that should exist before routes reference it via `expose_headers: "X-Correlation-Id"`.

**Important:** The `expose_headers: "X-Correlation-Id"` in each CORS plugin config (created in 05-02 and 05-03) must already list the exact header name that the request-id plugin generates. Since we set `header_name: "X-Correlation-Id"` in the global rule and `expose_headers: "X-Correlation-Id"` in each CORS config, these are already consistent. **No changes needed to existing CORS configs.**

## Troubleshooting

### APISIX returns 503 Service Unavailable
Upstream not reachable. Verify:
1. API projects are running (check Aspire Dashboard)
2. `host.docker.internal` resolves from within the APISIX container
3. Ports 5000 (dragonball) and 5002 (music) are accessible

### APISIX returns 404 on routes
Routes not created. Verify:
1. etcd is running and APISIX can connect
2. Init script executed without errors (check container logs)
3. Admin API is reachable: `docker exec <apisix-container> curl -s http://127.0.0.1:9180/apisix/admin/routes -H 'X-API-KEY: edd1c9f034335f136f87ad84b625c8f1'`

### OIDC returns 401 even with valid token
1. Check Keycloak is running: `curl -s http://localhost:8080/realms/OpenCode/.well-known/openid-configuration`
2. Verify client_secret matches Keycloak client config
3. Verify bearer token is not expired
4. Check APISIX container logs for OIDC plugin errors
5. Verify `discovery` URL uses `keycloak:8080` (container DNS) not `host.docker.internal:8080`

### CORS preflight fails
1. Verify origin is in the allow list (localhost:5173, etc.)
2. If using `allow_credential: true`, verify origins are explicit (not `*`)
3. Check that no .NET CORS middleware is also running (creates double headers)
