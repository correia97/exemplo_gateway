### Task 1: Add OIDC write routes to init-routes.sh
Append write-route creation blocks to `.planning/phases/05-apisix-gateway/init-routes.sh`, inserted after the GET route blocks and before the final "Route initialization complete" echo.

Add the following code to the script (insert before the final `echo "Route initialization complete."` line):

```bash
# === Create Write Routes (POST/PUT/DELETE — OIDC-protected) ===

echo "Creating write route: /api/dragonball/* (OIDC)"
curl -s -X PUT "$ADMIN_API/routes/dragonball-write" \
    -H "X-API-KEY: $ADMIN_KEY" \
    -H "Content-Type: application/json" \
    -d "$(cat <<EOF
{
    "uri": "/api/dragonball/*",
    "methods": ["POST", "PUT", "DELETE"],
    "strip_path": true,
    "upstream_id": "dragonball",
    "plugins": {
        "cors": {
            "allow_origins": "$CORS_ORIGINS",
            "allow_methods": "GET,POST,PUT,DELETE,OPTIONS",
            "allow_headers": "*",
            "expose_headers": "X-Correlation-Id",
            "allow_credential": true,
            "max_age": 86400
        },
        "openid-connect": {
            "client_id": "dragonball-api",
            "client_secret": "$CLIENT_SECRET_DRAGONBALL",
            "discovery": "http://keycloak:8080/realms/OpenCode/.well-known/openid-configuration",
            "bearer_only": true,
            "ssl_verify": false,
            "realm": "opencode"
        }
    }
}
EOF
)"

echo "Creating write route: /api/music/* (OIDC)"
curl -s -X PUT "$ADMIN_API/routes/music-write" \
    -H "X-API-KEY: $ADMIN_KEY" \
    -H "Content-Type: application/json" \
    -d "$(cat <<EOF
{
    "uri": "/api/music/*",
    "methods": ["POST", "PUT", "DELETE"],
    "strip_path": true,
    "upstream_id": "music",
    "plugins": {
        "cors": {
            "allow_origins": "$CORS_ORIGINS",
            "allow_methods": "GET,POST,PUT,DELETE,OPTIONS",
            "allow_headers": "*",
            "expose_headers": "X-Correlation-Id",
            "allow_credential": true,
            "max_age": 86400
        },
        "openid-connect": {
            "client_id": "music-api",
            "client_secret": "$CLIENT_SECRET_MUSIC",
            "discovery": "http://keycloak:8080/realms/OpenCode/.well-known/openid-configuration",
            "bearer_only": true,
            "ssl_verify": false,
            "realm": "opencode"
        }
    }
}
EOF
)"
```

**Key details:**
- `methods: ["POST", "PUT", "DELETE"]` — write-only, GET is excluded (already handled by separate route)
- `bearer_only: true` — no redirect to Keycloak login page, validates Bearer JWT token only
- `discovery` — uses `keycloak:8080` (Aspire container DNS within the app model network) — verified that Aspire's AddContainer resolves container service names via internal DNS
- `client_secret` — references environment variable `$CLIENT_SECRET_DRAGONBALL` / `$CLIENT_SECRET_MUSIC` set in Program.cs
- CORS plugin included on write routes too (browsers send CORS preflight for cross-origin POST/PUT/DELETE)
- `strip_path: true` — same as GET routes, upstream gets clean paths

### Task 2: Verify CORS preflight handling
The CORS plugin in Kong automatically handles OPTIONS preflight requests by returning CORS headers without forwarding to the upstream. This needs to work for both route groups.

**Verification note:** Since the CORS plugin is on both GET and write routes with the same configuration, any OPTIONS request matching either URI pattern will be handled by whichever route matches first. Kong 3.x matches routes by `uri` pattern, and OPTIONS is listed in `allow_methods` on both routes. For OPTIONS handling, the CORS plugin intercepts before upstream dispatch.

To ensure Kong handles OPTIONS before consulting methods, add the `_meta.filter_func` or verify that the CORS plugin's built-in OPTIONS handling works correctly. In Kong 3.x, the CORS plugin automatically handles OPTIONS requests when present on a route — it returns CORS headers without checking the `methods` filter. So OPTIONS will be handled regardless of which route matches.
